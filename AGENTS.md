# AGENTS.md

이 문서는 굿애프터눈 프로젝트에서 작업하는 에이전트와 개발자를 위한 공통 지침입니다. `GEMINI.md`는 Gemini용 harness 인덱스로 유지하고, 이 파일은 도구와 모델에 상관없이 적용되는 작업 기준으로 사용합니다.

## 작업 언어

- 모든 응답, 문서, 주석은 한국어 존댓말을 기본으로 합니다.
- 파일 인코딩은 UTF-8을 유지합니다.
- 사용자를 향한 설명은 짧고 구체적으로 씁니다.

## 프로젝트 미션

굿애프터눈은 공부처럼 느껴지지 않는 경제 개념 잠금해제 서비스입니다. 핵심 철학은 `Easy to learn, Hard to master`입니다.

- Easy to learn: 첫 진입은 장소, 행동, 짧은 로그로 구성합니다.
- Hard to master: 후반부는 복합 판단, 지연 피드백, 실제 경제 시나리오 분석으로 깊이를 만듭니다.
- 철 티어 이전에는 `Good Afternoon.` 이름의 의미를 직접 설명하지 않습니다.

## 게임 정체성

굿애프터눈은 2D 마을 탐험형 경제 선택 게임입니다. 경제 지식을 맞히는 앱이 아니라, 작은 마을을 돌아다니며 선택하고, 그 선택의 결과가 길, 장소, 관계, 기회, 위험으로 돌아오는 게임이어야 합니다.

- 기본 감각은 `Stardew Valley`/`Animal Crossing`식 마을 생활과 `Pokemon` 초기 지역 잠금해제 구조에 가깝습니다.
- 선택의 압축감은 `Reigns`, 후반 판단의 긴장은 `Papers, Please`를 참고합니다.
- 보상 감각은 `Zelda`식 길 열림, 새 장소 발견, 새 상호작용 해금에 가깝습니다.
- 전투, 수집, 꾸미기보다 장소 탐색, 짧은 상황, 선택 결과, 잠금해제가 핵심 재미입니다.

## 재미 우선 원칙

요구사항을 그대로 구현하기 전에, 먼저 이것이 게임으로 재미있는지 판단합니다. 기능은 다음 중 최소 하나를 강화해야 합니다.

- 탐색: 새 장소, 새 길, 숨은 입구, 다시 가보고 싶은 이유를 만든다.
- 선택: 정답 클릭이 아니라 상황 속 행동을 고르게 한다.
- 결과: 선택이 즉시 또는 나중에 마을 상태, NPC 반응, 접근 가능한 장소에 영향을 준다.
- 호기심: 설명보다 먼저 “저건 뭐지?”라는 질문을 만든다.
- 숙련: 같은 선택도 맥락에 따라 다르게 판단하게 만든다.

다음과 같은 구현은 다시 설계합니다.

- 요구사항은 만족하지만 카드, 목록, 체크리스트, 설명문만 늘어나는 구현
- 경제 용어를 먼저 설명하고 나중에 게임 화면을 붙이는 구현
- 버튼을 누르면 완료 처리만 되고 마을이나 선택지가 달라지지 않는 구현
- 점수, 완료율, 정답 수를 핵심 보상으로 앞세우는 구현
- Pre-Cyan에서 후반 마스터리의 복잡한 판단을 강요하는 구현

## 핵심 게임 루프

모든 플레이 기능은 가능한 한 아래 루프를 따릅니다.

1. 장소를 본다.
2. 짧은 상황을 만난다.
3. 행동을 선택한다.
4. 짧은 반응을 받는다.
5. 길, 장소, 관계, 힌트, 위험 중 하나가 변한다.
6. 나중에 이전 선택의 의미가 다시 돌아온다.

새 기능을 제안하거나 구현할 때는 이 루프에서 어느 단계를 강화하는지 먼저 밝힙니다. 루프에 들어가지 않는 기능은 설정, 저장, 접근성, 빌드 안정성처럼 게임을 지탱하는 작업일 때만 허용합니다.

## 문서 우선순위

1. `PROJECT_CONTEXT.md`: 서비스 정의와 확정된 기획 불변량
2. `docs/ROADMAP.md`: 개발 순서와 다음 구현 후보
3. `DESIGN.md`: Good Afternoon 디자인 기준
4. `harness/30_mastery_spec.md`: 마스터리 레이어 상세 설계
5. `harness/`: 검증, 무결성, SOP 세부 규칙
6. `README.md`: 새 개발자가 보는 실행 안내

작업 전에는 관련 문서를 먼저 읽고, 문서 간 충돌이 있으면 `PROJECT_CONTEXT.md`의 확정 구조를 우선합니다.

## 개발 원칙

- 현재 구현의 소스 오브 트루스는 `src/`입니다.
- Pre-Cyan 기준 구현은 `src/pre-cyan-village/`에서 관리합니다.
- Vite + TypeScript를 기준으로 개발하고, `npm run dev`, `npm test`, `npm run build` 흐름을 사용합니다.
- `dist/`는 빌드 출력물이며 커밋하지 않습니다.
- 이전 `app/` 경로는 새 구현의 기준선이 아닙니다. 필요한 경우 과거 구조 확인 용도로만 봅니다.
- 파일은 책임이 보이도록 작게 나눕니다.
- 기존 루트 HTML 데모는 참고 자료입니다. 새 구현의 소스로 복사하거나 덮어쓰지 않습니다.
- 새 기능은 `src/` 하위에 독립적인 TypeScript 모듈 또는 앱 슬라이스로 둡니다.
- 새 플레이 기능은 UI 컴포넌트가 아니라 게임 장면, 상태 변화, 선택 결과를 중심으로 설계합니다.
- 데이터 구조는 장소, 연결, 상태, 선택, 결과가 드러나도록 잡습니다. 단순 화면 렌더링용 배열에 게임 규칙을 숨기지 않습니다.
- "요구사항 구현 완료"만으로 끝내지 않습니다. 플레이어가 왜 다음 행동을 하고 싶은지까지 확인합니다.

## 디자인 원칙

- 따뜻한 연크림 캔버스, 차콜 텍스트, 브론즈 골드 포인트를 기본으로 합니다.
- Pre-Cyan 첫 경험은 카드 체크리스트가 아니라 노드와 길이 있는 마을 지도입니다.
- 긴 설명, 점수, 학습 완료 문구는 첫 경험 전면에 두지 않습니다.
- 보상은 숫자보다 “길이 열렸다”는 감각으로 표현합니다.
- 복권방 뒤 어두운 골목은 투자 브랜치의 숨은 씨앗으로 유지합니다.

## 작업 루프

모든 변경은 다음 순서로 진행합니다.

1. Research: 현재 문서와 코드를 읽습니다.
2. Game Check: 이 변경이 탐색, 선택, 결과, 호기심, 숙련 중 무엇을 강화하는지 정합니다.
3. Strategy: 바꿀 파일과 검증 기준을 정합니다.
4. Execution: 작고 명확한 변경을 적용합니다.
5. Validation: 문자열 스캔, 브라우저 스모크 테스트, 모바일 뷰포트 점검 중 해당하는 검증을 수행합니다.
6. Fun Review: 플레이어 관점에서 다음 행동을 하고 싶어지는지 점검합니다.

## 검증 기준

문서 변경:
- 오래된 미래형 설명이 남지 않았는지 확인합니다.
- 새 문서가 README 또는 관련 인덱스에서 발견 가능한지 확인합니다.
- `PROJECT_CONTEXT.md`, `docs/ROADMAP.md`, `DESIGN.md`가 서로 충돌하지 않는지 확인합니다.

프론트엔드 변경:
- 브라우저 콘솔 오류가 없어야 합니다.
- 모바일 390x844 기준으로 가로 스크롤과 텍스트 넘침이 없어야 합니다.
- 저장 상태가 새로고침 후 유지되고, 깨진 저장 데이터는 안전하게 초기화되어야 합니다.
- Pre-Cyan 첫 경험에 `필수 기초 단어 놀이터`, `이해함`, `점수`, `학습 완료` 같은 표현을 넣지 않습니다.
- 새 플레이 기능은 장소, 선택, 반응, 상태 변화 중 최소 2개를 화면에서 확인할 수 있어야 합니다.
- 사용자가 다음에 눌러보고 싶은 대상이 화면에 보여야 합니다.
- 보상은 가능하면 숫자보다 길 열림, 장소 변화, NPC 반응, 짧은 로그로 확인되어야 합니다.

게임성 점검:
- 이 기능이 없을 때보다 마을을 더 탐색하고 싶어지는지 확인합니다.
- 선택지가 단순 정답/오답인지, 상황 판단인지 확인합니다.
- 플레이어의 행동이 화면 또는 저장 상태에 흔적을 남기는지 확인합니다.
- 같은 UI를 카드 목록으로 바꿔도 의미가 거의 같다면 다시 설계합니다.

## 금지 사항

- 사용자가 요청하지 않은 대규모 프레임워크 전환
- 루트 HTML 데모를 새 구현 소스로 취급하는 것
- 철 티어 이전에 앱 이름의 의미를 설명하는 것
- Pre-Cyan에 마스터리 레이어의 복잡도를 끌어오는 것
- 게임 루프 없이 학습 콘텐츠만 쌓는 것
- 재미 검토 없이 요구사항 충족만 보고 완료 처리하는 것
- 사용자 변경사항을 임의로 되돌리는 것
