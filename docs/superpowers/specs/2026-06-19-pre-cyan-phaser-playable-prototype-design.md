# Pre-Cyan Phaser Playable Prototype Design

> Execution note: user direction changed after this design. Phaser-only became the implemented Pre-Cyan path, and the DOM fallback/removal gate steps were collapsed into implementation. See `2026-06-19-pre-cyan-phaser-dom-removal-gate.md` for the current gate record.

## Summary

Pre-Cyan 첫 경험을 DOM 기반 선택 UI에서 Phaser 기반 2D 플레이어블 경험으로 전환한다. 목표는 웹 UI를 게임처럼 꾸미는 것이 아니라, 웹에서 돌아가는 실제 2D 게임 런타임을 도입해 `내 방에서 시작 → 문으로 나감 → 작은 마을을 직접 걷기 → 장소 근처에서 상호작용 → 하루 기록이 방에 남음`을 플레이 감각으로 전달하는 것이다.

첫 구현은 완성형 게임이 아니라 플레이어블 프로토타입이다. 실행 과정에서 사용자 방향이 Phaser-only로 바뀌었고, DOM 런타임은 Pre-Cyan entry/source에서 제거됐다.

## Product Goals

- Pre-Cyan을 `진짜 게임처럼 조작하는 첫 정착 경험`으로 만든다.
- 캐릭터 직접 이동과 명시적 상호작용으로 사용자가 선택했다는 감각을 만든다.
- 탑다운 조작의 단순함과 아이소메트릭 보드게임풍 미감을 섞는다.
- 기존 `domain/data.ts`, `domain/outing.ts`, `domain/state.ts`의 규칙과 저장 흐름을 최대한 재사용한다.
- GitHub Pages 배포를 유지한다.
- Phaser 구현을 Pre-Cyan 기본 진입점으로 둔다.

## Non-Goals

- 첫 버전에서 완전한 타일맵 에디터 파이프라인을 도입하지 않는다.
- 첫 버전에서 NPC 대화 시스템을 만들지 않는다.
- 첫 버전에서 자유도 높은 재화, 인벤토리, 스태미나 시뮬레이션을 만들지 않는다.
- 첫 버전에서 복권방, 어두운 틈, Cyan 입구를 완전 상호작용 콘텐츠로 열지 않는다.
- 제거된 DOM Pre-Cyan 구현을 fallback으로 되살리지 않는다.
- Pre-Cyan 첫 경험에 점수, 정답/오답, 학습 완료, 체크리스트 느낌을 넣지 않는다.

## Chosen Direction

### Runtime

Phaser를 2D 게임 런타임으로 사용한다. Vite + TypeScript + GitHub Pages 배포 흐름은 유지한다.

Phaser가 담당할 영역:

- 씬 전환
- 캐릭터 이동
- 카메라와 화면 스케일
- 충돌 또는 이동 제한
- 근접 상호작용 판정
- 페이드, 경로선, 장소 하이라이트 같은 게임 연출

기존 도메인 레이어가 담당할 영역:

- 현재 외출 ID
- 상호작용 순서 기록
- 장소 반응 결정
- 돈독 같은 숨은 플래그
- Cyan 입구 흔적 조건
- localStorage 저장/복구

### Camera And Map Style

실제 이동과 좌표계는 탑다운으로 둔다. 사용자는 WASD/방향키 또는 모바일 이동 패드로 위, 아래, 왼쪽, 오른쪽 이동을 이해하면 된다.

화면 표현은 아이소 감성을 섞는다.

- 바닥 패턴은 살짝 비스듬한 길과 그림자로 미니어처 지도 느낌을 준다.
- 건물은 탑다운 충돌 박스 위에 대각선 지붕, 그림자, 간판을 얹는다.
- 장소 노드는 얇은 브론즈 골드 선과 은은한 빛으로 표현한다.
- 길이 열릴 때는 현실 도로가 생기는 것이 아니라, 지도 위 경로선이 켜지는 느낌을 준다.

이 절충안은 완전 아이소메트릭보다 구현 난도가 낮고, 완전 탑다운보다 Good Afternoon의 `길이 열렸다`는 감각을 더 잘 전달한다.

## Scene Model

### RoomScene

`RoomScene`은 시작 공간이자 복귀 허브다. 첫 버전에서는 상호작용 가능한 미니 허브까지 만들지 않고, 허브의 성격을 시각적으로 먼저 확립한다.

초기 구성:

- 작은 방
- 플레이어 시작 위치
- 문
- 작은 단말
- 기록 종이 또는 벽면 흔적

초기 동작:

- 사용자는 방 안에서 캐릭터를 움직일 수 있다.
- 문 근처에 가면 상호작용 힌트가 뜬다.
- PC에서는 `E`, 모바일에서는 상호작용 버튼으로 문을 사용한다.
- 문을 사용하면 짧은 페이드 후 `VillageScene`으로 전환한다.

복귀 후 변화:

- 마지막 외출의 기록이 방에 남는다.
- 단말 로그가 짧게 갱신된다.
- 필요한 경우 기록 종이, 방 안 빛, 벽면 흔적이 바뀐다.

확장 예정:

- 단말 열람
- 기록 다시 보기
- 물건 조사
- 티어별 방 변화
- K 단계 회고 허브

### VillageScene

`VillageScene`은 작은 단일 화면 마을이다. 첫 버전에서는 큰 월드 탐험보다 짧은 플레이 감각 검증이 우선이다.

배치 요소:

- 은행
- 편의점
- 버스정류장
- 게시판
- 복권방
- 어두운 틈
- Cyan 입구 흔적
- 방으로 이어지는 귀가 경로 암시

핵심 상호작용 장소:

- 은행
- 편의점
- 버스정류장
- 게시판

배경/떡밥 장소:

- 복권방
- 어두운 틈
- Cyan 입구 흔적

복권방, 어두운 틈, Cyan 입구는 처음부터 마을 안에 보인다. 다만 첫 프로토타입에서는 핵심 진행을 담당하지 않는다. 가까이 가면 빛, 노이즈, 색 변화, 아주 짧은 단말 반응 정도만 줄 수 있다. 초반 사용자가 눌러야 할 대상은 많아 보이지 않게 유지한다.

## Interaction Model

상호작용은 `근처 접근 + 명시적 입력` 방식이다.

PC:

- 이동: WASD 또는 방향키
- 상호작용: `E`

모바일:

- 이동: 화면 하단 이동 패드 또는 터치 드래그
- 상호작용: 별도 버튼

장소 근처에 들어오면 라벨, 은은한 빛, 짧은 힌트가 뜬다. 사용자가 직접 상호작용 입력을 해야 반응이 발생한다. 접근 즉시 이벤트를 발동하지 않는다.

이 방식은 사용자가 `내가 선택했다`는 감각을 갖게 하고, 실수로 중요한 이벤트가 발생하는 일을 줄인다.

## Outing Loop In Game Form

기존 외출 루프는 유지하되, 화면에는 자유 탐험처럼 보이게 한다.

내부 규칙:

- 한 번의 외출은 핵심 상호작용 3회를 기록한다.
- 핵심 장소는 현재 외출 데이터와 기존 outing 도메인 로직에 연결된다.
- 같은 장소라도 이전 상호작용 순서와 상태에 따라 반응이 달라질 수 있다.
- 3번째 핵심 상호작용 후 외출 결과가 정리된다.

플레이어에게 보이는 흐름:

1. 방에서 문을 사용해 마을로 나간다.
2. 마을을 자유롭게 걸어다닌다.
3. 장소 근처에서 `E` 또는 상호작용 버튼을 누른다.
4. 짧은 단말 로그와 장소 반응이 나온다.
5. 3번째 핵심 상호작용 후 귀가 경로가 열린 것처럼 보인다.
6. 짧은 페이드 후 자동으로 방으로 복귀한다.
7. 방에 오늘 기록이 남는다.

복귀는 실제로는 자동이다. 다만 연출은 `집으로 돌아갈 길이 켜졌다`처럼 보여준다. 첫 프로토타입에서 플레이어에게 별도 귀가 조작을 요구하지 않는다.

## Tone And UI

첫 마을 분위기는 `따뜻하지만 살짝 수상한 오후 동네`다.

- 은행, 편의점, 정류장, 게시판은 생활감 있게 보인다.
- 복권방은 간판 불빛이나 문틈이 조금 과하게 반짝인다.
- 어두운 틈은 가까이 가면 약한 노이즈나 색 변화가 있다.
- Cyan 입구 흔적은 멀리서만 희미하게 보인다.
- 초반에는 아무도 의미를 직접 설명하지 않는다.

UI는 최소 HUD만 둔다.

- 작은 단말 아이콘
- 짧은 안내 한 줄
- 장소 근처에서만 뜨는 상호작용 힌트
- 모바일 이동 패드와 상호작용 버튼

표시하지 않을 것:

- 방문 기록 숫자
- 체크리스트
- 점수
- 학습 완료 표현
- 경제 개념 설명 카드

텍스트는 짧은 단말 로그 중심이다. NPC 대화는 후속 확장으로 미룬다.

## Visual Asset Strategy

첫 프로토타입은 직접 만든 미니 SVG 또는 PNG 자산을 사용한다.

필요 자산:

- 작은 플레이어 캐릭터
- 방 바닥과 문
- 단말
- 기록 종이 또는 벽면 흔적
- 은행
- 편의점
- 버스정류장
- 게시판
- 복권방
- 어두운 틈
- Cyan 입구 흔적
- 경로선과 노드 빛

처음부터 스프라이트시트와 타일맵 전체 파이프라인을 강제하지 않는다. 필요한 최소 자산으로 게임 감각을 검증하고, 이후 반복 제작이 필요해지면 타일맵 또는 스프라이트시트 기반으로 확장한다.

## Architecture

제안 구조:

```text
src/pre-cyan-village/
  domain/
    data.ts
    outing.ts
    state.ts
  game/
    main-game.ts
    scenes/
      RoomScene.ts
      VillageScene.ts
    objects/
      Player.ts
      InteractionPrompt.ts
      DeviceHud.ts
      PathGlow.ts
    adapters/
      outing-session.ts
      storage-adapter.ts
  assets/
    player.svg
    room.svg
    bank.svg
    store.svg
    bus-stop.svg
    board.svg
    lottery.svg
    cyan-trace.svg
```

`domain/`은 Phaser에 의존하지 않는다. `game/adapters/`가 Phaser 씬과 도메인 로직 사이를 연결한다.

예상 데이터 흐름:

1. 앱 시작 시 `loadState`로 상태를 복구한다.
2. Phaser 게임을 시작하고 `RoomScene`에 상태를 주입한다.
3. 문 상호작용 시 `startOuting` 또는 현재 외출 초기화 로직을 호출한다.
4. `VillageScene`에서 장소 상호작용 시 도메인 어댑터가 `selectHotspot`을 호출한다.
5. 상태가 바뀌면 `saveState`를 호출한다.
6. 3번째 핵심 상호작용 후 씬이 귀가 연출을 실행하고 `RoomScene`으로 돌아간다.
7. 방 씬은 상태를 읽어 기록/빛/단말 로그를 갱신한다.

## Migration Plan

아래 전환 순서는 초기 설계의 단계적 경로다. 실제 실행은 사용자 결정에 따라 Phaser-only로 수렴했고, DOM 제거 gate는 구현 안에 합쳐졌다.

1. Phaser 런타임을 추가한다.
2. 도메인 로직을 Phaser 씬에서 호출할 수 있도록 얇은 어댑터를 만든다.
3. `RoomScene`과 `VillageScene`으로 최소 플레이 루프를 구현한다.
4. Playwright로 `/good-afternoon/` 배포 경로에서 Phaser 캔버스, 이동, 상호작용, 복귀를 검증한다.
5. 기존 DOM 구현과 동일한 저장/복구 조건을 만족하는지 확인한다.
6. Phaser 런타임이 Pre-Cyan 첫 경험의 주 실행 경로가 된 상태를 gate 문서로 기록한다.

DOM 제거 조건:

- Phaser 버전에서 방 시작, 마을 진입, 3회 핵심 상호작용, 자동 복귀가 작동한다.
- 저장 상태가 새로고침 후 복구된다.
- 깨진 저장 데이터가 안전하게 초기화된다.
- 모바일 390x844에서 가로 스크롤과 텍스트 넘침이 없다.
- GitHub Pages base 경로에서 JS/CSS/assets가 404 없이 로드된다.
- 금지 문자열이 Pre-Cyan 런타임에 없다.

## Testing And Verification

도메인 테스트:

- 기존 `outing.ts`, `state.ts` 테스트를 유지한다.
- Phaser 의존 없이 상호작용 어댑터의 상태 전이를 테스트한다.

정적 계약 테스트:

- Vite entry가 Phaser 진입점을 가진다.
- GitHub Pages base path가 유지된다.
- 금지 문자열이 런타임 소스에 없다.

브라우저 스모크:

- 데스크톱 뷰포트에서 캔버스가 비어 있지 않다.
- 모바일 390x844에서 가로 overflow가 없다.
- 방에서 문 상호작용 후 마을 씬으로 전환된다.
- 마을에서 핵심 장소 3곳 상호작용 후 방으로 복귀한다.
- 콘솔 오류와 asset 404가 없다.

수동 확인:

- 첫 30초 안에 사용자가 어디로 가야 할지 감을 잡는다.
- UI가 체크리스트나 학습 앱처럼 보이지 않는다.
- 복권방/어두운 틈/Cyan 입구가 강하게 설명되지 않고 배경 떡밥으로 남는다.

## Implementation Defaults

아래 기본값으로 첫 구현 계획을 작성한다. 구현 중 명확한 기술적 문제가 발견될 때만 별도 변경 제안을 만든다.

- Phaser는 3.x 계열을 사용한다.
- 모바일 이동은 첫 버전에서 고정형 화면 하단 이동 패드로 구현한다.
- 직접 제작 자산은 SVG 원본을 저장소에 두고, Phaser에서 이미지 텍스처로 로드한다.
- Phaser 런타임은 기본 진입점이다.
- 방/마을/저장/모바일/배포 검증은 Phaser 기본 진입점 기준으로 수행한다.
- 제거된 DOM 런타임은 fallback으로 유지하지 않는다.
- 첫 플레이어 캐릭터는 4방향 상태를 가진다. 첫 버전에서는 방향별 정지 프레임만 있어도 된다.
